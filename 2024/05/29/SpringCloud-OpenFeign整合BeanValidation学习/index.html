SpringCloud-OpenFeign整合BeanValidation学习

    测试用部分代码如下

    只是这样在feign interface的方法上加上@Valid注解的话，去调用接口，可以看到bean validation是没起作用的。归根结底是因为SpringCloud-OpenFeign使用的SynchronousMethodHandler没有像SpringMVC使用的InvocableHandlerMethod一样在处理方法参数时有参数校验的逻辑。

    所以如果想在SpringCloud-OpenFeign中整合Bean Validation还得想想办法，下面是我整理的几种方法

一：Spring提供的@Validated注解

    然后再调用Controller接口（调用堆栈如下图一所示，发现此时的feignTestService）变成了JdkDynamicAopProxy，然后才是ReflectiveFeign$FeignInvocationHandler，可以对比下图二看下两者的区别。

    看到走到了 JdkDynamicAopProxy 里面

    一直debug到 MethodValidationInterceptor 的 invoke方法

    可以看到在feign接口上添加@Validated注解后实际上是将feign实例进行了动态代理，在feign方法执行前后由专门的拦截器 MethodValidationInterceptor 进行参数校验。

   下面研究下Spring是如何根据@Validated注解对feign client实例进行动态代理的，emmm，主要涉及三个类AbstractAdvisingBeanPostProcessor、AbstractBeanFactoryAwareAdvisingPostProcessor、MethodValidationPostProcessor，还有一个FilteredMethodValidationPostProcessor可以注入一些MethodValidationExcludeFilter标注校验时排除哪些类，感兴趣的读者可以自行研究

二：实现 RequestInterceptor 接口 使用请求模板上的方法来检查查询和值

    实现原理：SynchronousMethodHandler（Class） => invoke（Method） => executeAndDecode（Method） => targetRequest（Method）

    这里会调用每个拦截器的apply方法以便在远程方法调用之前对 RequestTemplate 进行自定义操作（注意，这里只能处理请求头和查询字符串参数）

三：实现Encoder 接口来校验对象

    实现原理：SynchronousMethodHandler（Class） => buildTemplateFromArgs.create（Method）=> resolve（Method）=> encoder.encode（Method）

    其中object是参数，bodyType是参数类型

    这里我们拿到了参数，就可以通过Validator来对请求体进行参数校验了（注意：RequestTemplate有一个属性为MethodMetadata methodMetadata，能拿到方法信息，可以据此判断参数是否需要校验）

四：通过为参数提供一个自定义的Expander 来验证单个参数

    在Spring Cloud OpenFeign中，Expander接口允许你自定义参数的展开逻辑，这对于处理复杂的URL模板参数非常有用。例如，如果你需要将某个复杂对象转换为URL路径的一部分，或者对查询参数进行特殊处理，就可以实现自定义的Expander。

    使用示例，可以在参数expand前进行校验

    深入使用可以结合AnnotatedParameterProcessor，可以参考（validation - How to validate request parameters on feign client - Stack Overflow）和（spring - How to custom @FeignClient Expander to convert param？ - Stack Overflow）
